# 智柚项目 UI / 功能修改必读清单

## 适用范围
本清单适用于 `UI设计` 项目内所有与以下内容有关的修改：
- UI 页面调整
- 响应式布局调整
- 学员 / 工作人员视角相关修改
- 路由、导航、模块入口调整
- 功能运行逻辑改动
- 文案去重、模块合并、内容映射

本清单是**每次动手前必读规则**，优先级高于临时发挥。

---

## 一、项目总原则

### 1.1 先保护，再优化
本项目当前阶段的首要目标不是“快速重做”，而是：
- 保护现有 UI 不被改崩
- 保护双视角结构不被误伤
- 保护响应式适配不被破坏
- 保护现有可运行链路不进入反复修复循环

### 1.2 业务文档不是代码改动清单
录音整理文档、岗位流程文档、优化方案文档，**只能用于理解需求**，不能直接按文档顺序对页面做逐条硬改。

任何业务材料在进入代码修改前，必须先转换为：
- 影响页面列表
- 不影响页面列表
- 删除项清单
- 保留项清单
- 学员侧归属 / 工作人员侧归属
- 是否涉及响应式结构

### 1.3 默认保守策略
若存在不确定性，默认：
- 不删除
- 不合并
- 不重排
- 不覆盖旧逻辑
- 不改共享骨架
- 不碰高风险文件

不确定时先停，不允许靠“边改边试”碰运气。

### 1.4 执行说明只在聊天框反馈，不进入业务 UI
后续无论是完善、优化还是修改当前项目，完成任务后的说明只在聊天框告知用户，不得为了说明本次改动而在页面 UI 中新增介绍卡片、说明框、来源框、完成标准框、任务定位框等非业务必要展示。

尤其注意：
- 不把“我做了什么”写进页面
- 不把数据来源、挂载建议、完成标准作为额外展示卡片塞进课程页或首页
- 真实数据可以保留在数据字段中，但默认不新增可见说明区块
- 若用户只要求完成任务，页面只呈现业务功能本身

## 二、绝对禁止清单

### 2.1 禁止直接重做整页
禁止在没有拆清边界前，对现有页面执行以下行为：
- 整页推翻重写
- 大面积结构重排
- 直接替换整段 JSX 布局
- 为了业务调整顺手改容器层级

### 2.2 禁止一次改多个高耦合页面
单次修改禁止同时大改以下多个页面：
- `Layout.tsx`
- `routes.tsx`
- `Workbench.tsx`
- `AppContext.tsx`
- 任一带教 / 工作台核心页

规则：**一次只允许主改 1 个页面，最多连带 1 个数据或配置文件。**

### 2.3 禁止未审查就删除重复项
凡是“重复显示项”，在未完成判定前，禁止直接删除。必须先判断它属于哪一种：
- 视觉重复
- 数据重复
- 双身份下的有意重复
- 不同场景下的同概念展示

未判定前一律不可删。

### 2.4 禁止误伤学员视角
禁止把工作人员需求直接映射到学员侧页面。
尤其禁止：
- 为工作人员改动顺手删学员模块
- 将工作人员入口并入学员展示
- 将学员内容误当运营冗余而删除
- 让两个视角共用错误的数据展示语义

### 2.5 禁止改动一级导航和主链路
未经明确批准，禁止修改：
- 一级导航结构
- 顶部身份切换逻辑
- `/learning/*` 主链路
- 已成立的学员学习闭环
- 已成立的工作人员工作台骨架

### 2.6 禁止第一轮动响应式父容器
第一轮修改禁止改动以下高风险结构：
- 外层 `grid`
- 父级 `flex`
- `max-w-*`
- `md:*` / `lg:*` / `xl:*` 断点类
- `hidden md:*` / `md:hidden` 显隐策略
- 主区块间距、页头页体外边距、栅格列数

第一轮只允许改内容层、数据层、文案层，不允许顺手改布局骨架。

### 2.7 禁止硬删除旧逻辑
第一轮不允许直接硬删除原有内容，优先级必须是：
1. 标记弃用
2. 隐藏展示
3. 切换数据源
4. 临时开关隔离
5. 验证无影响后再删除

### 2.8 禁止边修边扩需求
一旦进入问题修复，不允许同时追加新优化需求。修复阶段只做：
- 定位原因
- 最小修复
- 回归验证

禁止“顺便一起优化”。

### 2.9 禁止未检索引用就改名 / 删除 / 移动共享代码
凡是涉及以下对象，禁止仅凭文件名或局部上下文直接修改：
- 共享组件
- 公共 Hook
- 类型定义
- 路由路径
- 导航 key
- 数据字段名
- `visualTokens` 等视觉 token

规则：**先全局检索引用，再决定能不能动。**

### 2.10 禁止用全局改动修局部问题
如果问题只出现在单个页面或单个模块，禁止优先改动以下全局区域：
- 全局样式文件
- 视觉 token
- 共享布局容器
- 公共组件默认样式
- 全局状态字段语义

局部问题必须先尝试局部修复，不能为了省事去改全局。

### 2.11 禁止项目级批量替换 / 批量格式化
未经明确确认，禁止执行以下高风险动作：
- 全项目批量搜索替换
- 批量重命名标题 / 路由 / 字段名
- 因局部需求触发整文件大重排
- 因格式化导致大量无业务意义 diff

本项目当前阶段更重视稳定，不接受“大范围看起来都顺手整理一下”的修改方式。

### 2.12 禁止凭“看起来没用”清理代码
禁止因为以下主观判断就直接删除：
- 看起来没引用
- 当前页面没显示
- 暂时没走到这个分支
- 文案似乎重复
- 某个数组项暂时没被点到

本项目存在双视角、条件渲染、共享骨架、阶段性影子能力，很多内容**可能是低频使用，不是无效代码**。

## 三、必须保护的核心区域

### 3.1 身份视角边界
本项目是**同一产品双视角**，不是两个独立系统。
因此必须明确：
- 学员视角和工作人员视角可以共用框架，但不能混淆展示目的
- 同名模块不等于同内容
- 共享骨架不等于共享语义
- 工作人员的运营目标不能反推修改学员体验

特别注意：学习身份中的“社区运营”和“运营管理者”不是两类独立用户，也不是两个并列学习视角，而是同一学习身份下的两个阶段：
1. 先学习“社区运营”阶段
2. 再学习“运营管理者”阶段

涉及身份选择、学习路径、阶段卡片、按钮文案、数据映射时，必须按“同一身份的前后阶段”处理，不得拆成两个互斥身份，也不得把后阶段误当成独立角色入口。

### 3.2 响应式布局基线
本项目已有成熟的自适应页面结构，必须保护：
- 移动端与桌面端显隐逻辑
- 卡片栅格换行关系
- 主内容宽度节奏
- 标题、卡片、列表的垂直节奏
- 已调好的整体字体、字号、行高基线

特别要求：**不得回退到过小字号，不得压缩阅读体验。**

### 3.3 品牌与命名
后续项目正式命名统一为：`智柚`。
禁止在正式页面中再次引入其他产品名作为主名称。

APP 图标资源使用：`图片/APP.png`。
未经明确要求，不替换品牌图标来源。

### 3.4 高风险文件
以下文件默认视为高风险文件，修改前必须单独说明影响范围：
- `src/app/components/Layout.tsx`
- `src/app/routes.tsx`
- `src/app/pages/Workbench.tsx`
- `src/app/context/AppContext.tsx`
- `src/app/lib/visualTokens.ts`

对上述文件的改动，必须先说明：
- 为什么必须改
- 不改会怎样
- 会影响哪些页面
- 明确不会影响哪些页面

### 3.5 共享样式与公共组件保护区
以下对象默认按“共享资产”处理：
- 通用卡片、列表、导航、页头、弹窗等公共组件
- 通用 hooks / utils / types
- 全局样式文件
- 视觉 token、主题色、字号、间距规则

规则如下：
- 只因单页需求，不得直接修改共享资产默认行为
- 若必须改共享资产，必须先确认受影响页面范围
- 若无法确认受影响范围，禁止直接动共享资产
- 能通过页面局部覆盖解决的，不优先动公共默认值

## 四、页面改动顺序规范

### 4.1 正确顺序
所有修改必须遵守以下顺序：
1. 先确认视角边界
2. 再确认改动对象归属
3. 再处理重复项判定
4. 再处理内容映射
5. 最后才允许动视觉结构

### 4.2 错误顺序
以下顺序禁止出现：
- 先删内容，再判断是否属于另一个视角
- 先改布局，再回头补业务映射
- 先重写页面，再回头修响应式
- 先动多个页面，再统一验收

### 4.3 单页闭环原则
一次修改只完成一个页面的完整闭环：
- 范围确认
- 具体修改
- 双视角检查
- 三尺寸检查
- 风险说明

当前页面未闭环，不进入下一页。

---

## 五、删除 / 合并 / 去重规范

### 5.1 删除前必须有清单
任何删除操作前，必须先列出：
- 删除项名称
- 所在页面
- 所属视角
- 删除原因
- 是否有替代项
- 是否影响其他入口 / 数据 / 跳转

### 5.2 业务重复 ≠ 代码重复
禁止把以下情况直接当作重复代码处理：
- 相同标题但服务不同身份
- 相同入口名但目标页不同
- 相似模块但用于不同决策场景
- 同一数据在不同页面承担不同任务

### 5.3 合并必须先看语义
若两个模块看似相似，必须先判断是否同时满足以下条件：
- 用户对象相同
- 使用场景相同
- 决策动作相同
- 指标口径相同
- 页面位置职责相同

只要有一项不同，禁止强行合并。

---

## 六、运营化改造规范

### 6.1 先映射对象，不先改壳子
涉及培训 / 销售 / 运营内容迁移时，优先做对象映射：
- 当前对象是谁
- 运营化后对象是谁
- 原指标对应什么新指标
- 原风险对应什么新风险
- 原动作对应什么新动作

在映射表未完成前，禁止直接重写页面。

### 6.2 先走影子验证，不直接覆盖
若新增社区运营、门店运营、小区运营等新视角能力，默认优先策略：
- 新入口
- 影子页
- 影子数据
- 验证通过后再考虑替换

禁止一上来覆盖原有培训 / 销售主线页面。

---

## 七、功能运行修改规范

### 7.1 功能修改优先最小化
任何功能运行逻辑调整，优先：
- 最小范围改动
- 局部替换
- 显式新增
- 保留原行为兜底

禁止大面积联动式改造。

### 7.2 不得随意改全局状态语义
未经明确批准，禁止随意改动：
- 身份判断逻辑
- 全局上下文字段语义
- 页面共享状态结构
- 影响多个页面的公共工具函数

### 7.3 先判定是 UI 问题还是数据问题
若页面显示异常，禁止直接改 UI。必须先判断：
- 是布局问题
- 是数据源问题
- 是条件渲染问题
- 是视角判断问题
- 是文案映射问题

没有定位前，禁止直接补丁式乱修。

### 7.4 修改前必须先检索引用
若本次改动涉及以下任一对象，必须先做全局引用检索：
- 组件名
- Hook 名
- 类型名
- 路由路径
- 导航 key
- 数据字段名
- 条件渲染标识
- 共享样式类来源

未检索引用前，禁止直接删除、改名、搬移。

### 7.5 修改前必须先留当前稳定基线
每次正式改动前，至少记录以下信息：
- 当前页面路由
- 当前视角
- 当前页面正常表现
- 当前页面异常点
- 需要检查的屏幕尺寸
- 本次改动后的预期结果

若连“当前稳定状态是什么”都说不清，则不允许进入修改。

### 7.6 局部问题必须优先局部修复
若问题只发生在单页、单区块、单组件：
- 先改页面局部
- 再改局部子组件
- 最后才考虑共享组件或全局样式

禁止跳过局部排查，直接改全局。

### 7.7 数据字段名与文案 key 禁止就地硬改
涉及 mock 数据、状态字段、映射字段、文案 key 时：
- 不得只改当前页面使用处
- 不得因为当前页面报错就临时改字段名
- 不得为了匹配新文案直接替换旧字段语义

字段或 key 如需改名，必须在完成引用检索后，成套调整并验证影响范围。

## 八、每次修改前必须回答的 10 个问题

1. 这次改动服务于哪个视角：学员 / 工作人员？
2. 是否会误伤另一个视角？
3. 是否涉及一级导航、主路由、身份切换？若涉及，立即停止。
4. 是否会碰到响应式父容器？若会，默认拆成更小改动。
5. 这次改动是内容层、数据层还是布局层？
6. 是否存在“看似重复但其实职责不同”的模块？
7. 本次是否真的需要删除？能否先隐藏或标记弃用？
8. 本次修改会影响哪些页面？明确列出。
9. 本次修改明确不会影响哪些页面？明确列出。
10. 如果失败，是否可以快速回退？若不能，先不要改。

如果以上任一问题无法明确回答，则本次修改不得开始。

---

## 九、每次输出修改方案时的固定格式

每次给出修改方案，必须包含以下 5 项：

### 9.1 修改目标
一句话说明这次只解决什么问题。

### 9.2 本次只改哪些文件
严格列出文件名，不允许模糊描述。

### 9.3 本次明确不改哪些区域
至少列出：
- 不改页面
- 不改视角
- 不改导航 / 路由 / 响应式骨架

### 9.4 风险点
写清楚最可能出问题的地方。

### 9.5 验收项
至少包含：
- 学员视角检查
- 工作人员视角检查
- 桌面端检查
- 中屏检查
- 小屏检查

### 9.6 回退点
必须明确写出：
- 若本次失败，回退到哪个稳定点
- 哪些内容属于可快速撤回范围
- 哪些内容一旦修改就要谨慎回滚

### 9.7 引用检索说明
若本次涉及共享代码、字段改名、路由或导航调整，必须补充：
- 已检索哪些关键字
- 发现了哪些关联文件
- 为什么确认这次改动不会误伤其他页面

## 十、验收红线

### 10.1 出现以下任一情况，视为改坏
- 学员视角内容被误删
- 工作人员视角入口错乱
- 页面在不同屏幕尺寸下出现明显错位
- 原本可运行链路中断
- 一级导航异常
- 路由跳转异常
- 同一模块在两个视角下语义混乱
- 为了删除重复项导致信息缺失
- 局部改动引发其他页面样式异常
- 为修单页问题导致共享组件表现变化
- 数据字段或文案 key 被改动后引发连锁显示异常
- 全局样式或 token 被调整后破坏既有排版基线


### 10.2 一旦触发红线，立即停止扩改
处理顺序必须是：
1. 停止继续改其他页面
2. 回到最后一个稳定点
3. 只修当前问题
4. 修复后重新过完整验收

---

## 十一、项目级硬性要求

### 11.1 不允许进入修复循环
禁止出现以下工作方式：
- 改一处崩一处
- 崩了先补丁再说
- 补丁引出新问题再继续补
- 多页同时处于半修复状态

必须保证：**任何时刻都至少有一个稳定版本。**

### 11.2 不允许把“快”放在“稳”前面
本项目优先级排序固定为：
1. 结构安全
2. 视角清晰
3. 响应式稳定
4. 业务正确
5. 视觉优化速度

### 11.3 不确定时，先做文档，不做代码
若需求仍模糊，应先输出：
- 影响清单
- 删除判定清单
- 视角边界图
- 页面映射表

而不是直接改代码。

---

## 十二、执行口令（每次修改前先默读）

> 先分视角，再定范围；先保结构，再动内容；先做映射，再谈优化；不能确认，宁可不改；不为追求速度破坏稳定性。

这句话是本项目所有 UI / 功能修改的默认执行准则。
